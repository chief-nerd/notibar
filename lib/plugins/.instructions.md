# Notibar Plugin Authoring Guide

How to create a new plugin for Notibar. Follow every step — missing any will leave the plugin non-functional.

## Architecture Overview

A plugin connects to an external service (e.g. GitHub, Jira, Slack) and returns notification counts + items. Plugins never throw exceptions — they return errors inside `NotificationSummary.withError(...)`.

## Step-by-step Checklist

### 1. Add `ServiceType` enum value

In `lib/models/account.dart`, add the new value to the `ServiceType` enum:

```dart
enum ServiceType { microsoft, github, jira, slack, teams, frappe, mattermost, yourService, custom }
```

Keep `custom` last. After changing the enum, run `dart run build_runner build` to regenerate `account.g.dart`.

### 2. Implement the plugin class

Create `lib/plugins/<service_name>/<service_name>_plugin.dart`. The class must extend `NotibarPlugin`.

Copy the template and rename: **[_template/your_service_plugin.dart](_template/your_service_plugin.dart)**

The template shows the full pattern: credential validation, HTTP requests, error handling, `parseSummary` delegation, and the required presentation methods (`supportedMetrics`, `formatMenuEntry`, `serviceLabel`, etc.).

### 3. Register the plugin in the Bloc

In `lib/bloc/notibar_bloc.dart`, add the import and register in the `plugins` map:

```dart
import '../plugins/your_service/your_service_plugin.dart';

// Inside the NotibarBloc constructor's default plugins map:
ServiceType.yourService: YourServicePlugin(),
```

### 4. No UI wiring needed!

Plugins are self-describing. The settings window and tray manager read all UI metadata (labels, icons, supported metrics, menu formatting) directly from the plugin class via:

- `serviceLabel` — human-readable name (e.g. 'Your Service')
- `serviceIcon` — Material icon for the settings UI
- `configFields` — config keys shown in the Add Account dialog
- `supportedMetrics` — list of `MetricDefinition` objects with IDs, labels, SF Symbols, Material icons, count logic, and filter logic
- `formatMenuEntry` — how to render a `NotificationItem` as a native macOS menu entry

**You do not need to edit `settings_window.dart` or `tray_manager.dart`** unless you have a truly custom UI need.

### 5. Write tests

#### a. Create a JSON fixture

Save a representative API response to `test/fixtures/<service>_response.json`.

#### b. Add a parsing test

Add a test to `test/plugins/plugin_parsing_test.dart`. See template: **[_template/your_service_plugin_test.dart](_template/your_service_plugin_test.dart)**

Example fixture: **[_template/your_service_response.json](_template/your_service_response.json)**

Run tests: `flutter test`

## Key Models Reference

### Account (what the plugin receives)
| Field | Type | Usage |
|---|---|---|
| `apiKey` | `String?` | Auth token (PAT, Bearer token, API key) |
| `endpoint` | `String?` | Base URL for self-hosted services |
| `config` | `Map<String, String>` | Extra settings (owner, repo, projectKey, clientId, etc.) |
| `pollingInterval` | `Duration` | How often the bloc refreshes (default 5 min) |

### NotificationItem (what the plugin returns)
| Field | Type | Required | Notes |
|---|---|---|---|
| `id` | `String` | yes | Unique within this plugin |
| `title` | `String` | yes | Main display text |
| `subtitle` | `String?` | no | Secondary text (repo name, project, channel) |
| `body` | `String?` | no | Preview body text |
| `timestamp` | `DateTime` | yes | Used for sorting |
| `actionUrl` | `String` | yes | Opened in browser on click |
| `isUnread` | `bool` | no | Default `false` |
| `isFlagged` | `bool` | no | Default `false` |
| `metadata` | `Map<String, dynamic>` | no | Service-specific data for display/filtering |

### NotificationSummary (the return value)
| Field | Type | Notes |
|---|---|---|
| `unreadCount` | `int` | Used by metrics with id `'unread'` |
| `flaggedCount` | `int` | Used by metrics with id `'flagged'` |
| `mentionCount` | `int` | Used by metrics with id `'mentions'` |
| `assignedIssuesCount` | `int` | Used by metrics with id `'assignedIssues'` |
| `assignedPRsCount` | `int` | Used by metrics with id `'assignedPRs'` |
| `reviewRequestsCount` | `int` | Used by metrics with id `'reviewRequests'` |
| `items` | `List<NotificationItem>` | Always required (can be `[]`) |
| `error` | `PluginError?` | Set only on failure |
| `refreshedAccount` | `Account?` | Set if the plugin refreshed an OAuth token |

## Error Handling Rules

- **Never throw** from `fetchNotifications`. Always return `NotificationSummary.withError(...)`.
- Use `PluginErrorType.authentication` for 401/403 or missing credentials.
- Use `PluginErrorType.network` for `SocketException` / connectivity issues.
- Use `PluginErrorType.unknown` for everything else.
- Wrap the entire HTTP logic in try/catch. Catch `SocketException` specifically before the generic `catch (e)`.

## HTTP Conventions

- Use `package:http` (already a dependency). Do not add dio or other HTTP clients.
- Set `.timeout(const Duration(seconds: 15))` on requests.
- Always check for 401/403 status codes and return auth errors.
- For services needing OAuth token refresh (like Microsoft), store the refreshed token in `NotificationSummary.refreshedAccount` — the bloc will persist it.

## Conventions

- Plugin file lives in `lib/plugins/<service_name>/<service_name>_plugin.dart`.
- Class name: `<ServiceName>Plugin` (e.g. `GithubPlugin`, `SlackPlugin`).
- One plugin class per file.
- Keep `parseSummary` pure (no HTTP, no side effects) so it's testable with fixtures.
- Extra named parameters on `parseSummary` (like `baseUrl`) are fine — the interface signature uses optional params.
